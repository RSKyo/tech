# 📌 welm-radio / audiomanager 当前设计记录（2026-XX）

## 1. 项目定位

- 项目名：**welm-radio**
    
- 当前子系统拆分：
    
    - **audiomanager（独立系统）**：音频文件管理 CMS
        
    - **radio（独立系统）**：音频播放 / 电台运行时系统
        

👉 两者关系：

> audiomanager = 内容管理（上游）  
> radio = 播放系统（下游）

---

## 2. audiomanager 设计结论

### ✔ 最终命名

- 统一模块名：**audiomanager**
    

---

### ✔ 目录结构（当前稳定版）

```text
audiomanager/
├── core/
│   └── audio.js        # 文件扫描 / meta / index 逻辑

├── api/
│   └── audio.js        # Express API（req/res）

├── server.js           # 启动入口（express + static + api）

└── ui/
    └── audio.html      # UI + 内联 JS（不拆 audio-ui.js）
```

---

## 3. UI 设计决策

### ✔ 已确认决策

- ❌ 不使用 `audio-ui.js`
    
- ✔ UI 全部写在 `audio.html` 中（inline script）
    

---

### ✔ UI 技术结构

```html
audio.html
  ├── DOM结构
  ├── fetch("/api/audio-list")
  ├── render(list)
  └── init()
```

---

## 4. Express 架构理解（关键认知）

### ✔ app/server 关系

- `app = express()` → 应用实例（application）
    
- `server.js` → 启动入口（listen + bootstrap）
    

---

### ✔ API 调用模型

```text
browser UI
   ↓ fetch("/api/audio-list")
express server
   ↓
api/audio.js
   ↓
core/audio.js
   ↓
filesystem
```

---

## 5. server.js 当前结构状态

- express static 提供 ui
    
- 提供 `/api/audio-list`
    
- 读取 config.radio.root
    
- 调用 buildAudioList(root)
    

👉 当前属于：

> “单文件轻后端（monolith stage）”

---

## 6. UI / API 分层原则（已确认）

### ✔ UI 层（browser）

- 只负责：
    
    - render
        
    - fetch API
        
    - event handling
        

---

### ✔ API 层（server/api）

- 只负责：
    
    - req/res
        
    - 调用 core
        

---

### ✔ Core 层

- 只负责：
    
    - filesystem
        
    - meta
        
    - index
        

---

## 7. 关键架构决策总结

### ✔ 已确认原则：

- audiomanager 和 radio **必须分离（不是子模块关系）**
    
- UI 不再拆文件（避免 audio-ui.js）
    
- 采用“极简单 HTML + 内联 JS”结构
    
- express 作为本地 HTTP runtime（类 Tomcat 认知成立但更轻）
    

---

## 8. 命名体系最终选择

- 项目：`welm-radio`
    
- 子系统：
    
    - `audiomanager`（CMS）
        
    - `radio`（runtime）
        

---

但仍保持：
