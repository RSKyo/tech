## welm-radio 设计演化记录（2026-07-02）

今天这轮讨论的主线，是从一个简单的“可视化页面管理音频文件”需求，逐步演化成了一个更稳定的 `web` 子系统结构。最重要的收获不是某几段代码，而是项目边界逐渐清晰了。

---

# 1. 最初目标：用 Web 页面管理音频文件

最早的目标很简单：

```txt
希望有一个可视化页面，用来查看和管理本地音频文件。
```

一开始的结构倾向是：

```txt
audiomanager/
├── server.js
├── core/
└── ui/
```

其中：

```txt
audiomanager/server.js   启动 Express 服务
audiomanager/ui          放页面
audiomanager/core        放音频扫描和 meta 逻辑
```

后来逐渐发现，`audiomanager` 只是一个 Web 子系统。未来还可能有：

```txt
audio-library
radio-player
dashboard
playlist-editor
```

所以不应该让这些 Web 子系统散落在项目根目录下。

---

# 2. 从 audiomanager 收敛到 web/audio-library

经过命名讨论，原来的 `audiomanager` 逐渐调整为：

```txt
web/audio-library
```

原因：

```txt
audiomanager 太长，也偏“管理器”
audio 太泛
audiolib 虽短，但略像内部缩写
audio-library 作为子 Web 项目名更正式、更清楚
```

最终倾向：

```txt
web/audio-library
```

而不是：

```txt
audiomanager
web/audio
web/audiolib
```

当前结构方向：

```txt
welm-radio
├── cmd/
├── infra/
├── web/
│   ├── server.js
│   ├── runner.js
│   └── audio-library/
│       ├── router.js
│       ├── service/
│       │   ├── audio.js
│       │   ├── file.js
│       │   └── meta.js
│       └── ui/
│           └── index.html
```

# 3. web/server.js：唯一 Web 服务入口

讨论过是否每个 Web 子系统都需要自己的 `server.js`。

结论是：**当前阶段不要每个子系统一个 server.js。**

原因：

```txt
每个子系统一个 server.js
=> 多端口、多进程、多套 ready/stop 管理
=> 过早复杂化
```

更合理的是：

```txt
web/server.js 是唯一 Web HTTP 服务入口
web/audio-library 是挂载在 web/server.js 下的一个子系统
```

所以现在：

```txt
web/server.js
```

负责：

```txt
1. 创建 express app
2. app.use(express.json())
3. 注册 ready 检测接口
4. 挂载 audio-library 静态目录
5. 挂载 audio-library router
6. app.listen(3000)
```

示例逻辑：

```js
app.get("/__ready", (req, res) => {
  res.json({ ok: true });
});

app.use(
  "/audio-library",
  express.static(path.join(__dirname, "audio-library/ui")),
);

app.use("/audio-library/api", audioLibraryRouter);

app.listen(3000, () => {
  console.log("welm-radio web server running: http://localhost:3000");
});
```

访问路径：

```txt
页面：
http://localhost:3000/audio-library/

API：
http://localhost:3000/audio-library/api/list

ready：
http://localhost:3000/__ready
```

# 4. server.js 是可执行入口，不应该 export

曾经想从 `web/server.js` 里导出 `serverPort`：

```js
import { serverPort } from "./server.js";
```

后来意识到这是错误方向。

原因：

```txt
ESM import 会执行被导入模块的顶层代码。
如果 runner.js import server.js，
server.js 里的 app.listen() 会被直接执行。
```

所以定下规则：

```txt
server.js 是直接执行的入口文件。
不要 import server.js。
不要从 server.js export 任何东西。
```

建议放在 `web/server.js` 顶部的提醒注释：

```js
/**
 * Executable web server entry.
 *
 * Do not import this file.
 * Do not export from this file.
 *
 * This file is meant to be executed directly by Node.js.
 */
```

关于 `3000` 是否要抽到 `config.js`，最后决定：

```txt
不抽。
server.js 写一次 3000。
runner.js 写一次 3000。
这是可以接受的重复。
```

原因是为了避免为了消灭一个默认值而引入额外配置文件。

# 5. web/runner.js：负责启动/停止 Web 服务

原来有一个：

```txt
infra/server.js
```

里面负责：

```txt
spawn 启动 server.js
轮询 HTTP ready
通过 lsof 停止端口进程
```

后来发现它已经明显是服务于 `web/server.js` 的能力，不再适合放在 `infra/`。

所以移动到：

```txt
web/runner.js
```

命名讨论过：

```txt
service.js     不好，容易和 web/audio-library/service/ 混淆
process.js     容易和 Node process 混淆
lifecycle.js   可行，但有点抽象
control.js     可行，但不如 runner 直观
runner.js      最合适
```

最终定为：

```txt
web/runner.js
```

职责：

```txt
web/server.js   Express 服务本体
web/runner.js   负责运行、检测、停止这个 Web 服务
```

后来进一步收敛：因为 web 结构固定，所以 `ensureWebServer()` 不再需要接收 `serverFile`，也不需要接收 ready URL。

也就是说，从：

```js
ensureNodeHttpService(serverFile, url, options)
```

逐步收敛为：

```js
ensureWebServer(options)
```

`web/runner.js` 内部固定知道：

```txt
serverPath = ./server.js
serverPort = 3000
readyUrl = http://localhost:3000/__ready
```

# 6. ready 检测从页面 URL 改成 /__ready

一开始 `ensureNodeHttpService()` 检测的是具体页面：

```txt
http://localhost:3000/audiomanager.html
```

后来改成：

```txt
http://localhost:3000/audio-library/
```

再后来发现，既然现在只有一个统一的 `web/server.js`，runner 只需要确认 Web server 是否启动，不应该绑定具体子系统页面。

所以最终建议：

```txt
web/server.js 提供 /__ready
web/runner.js 检测 /__ready
cmd/web.js 负责打开具体页面
```

边界变成：

```txt
web/runner.js
  确认 Web 服务活着

cmd/web.js
  打开 /audio-library/ 页面

web/server.js
  挂载所有 Web 子系统
```

# 7. cmd/audiomanager.js 改为 cmd/web.js

原来命令是针对单个 `audiomanager`：

```txt
cmd/audiomanager.js
welm-radio audiomanager ready
welm-radio audiomanager stop
```

结构演化后，`audiomanager` 已经不是独立服务，而是 `web/audio-library` 子系统。

所以命令文件改为：

```txt
cmd/web.js
```

命令形式也调整为：

```bash
welm-radio web audio-library
welm-radio web stop
```

现在 `cmd/web.js` 的职责：

```txt
1. ensureWebServer()
2. ensureChrome()
3. ensureChromePage("http://localhost:3000/audio-library/")
4. stopWebServer()
```

当前命令结构：

```js
export const WEB_COMMANDS = {
  "audio-library": {
    handler: cmd_web_audio_library,
  },

  stop: {
    handler: cmd_web_stop,
  },
};
```

命令语义：

```txt
welm-radio web audio-library
  启动 Web 服务并打开 Audio Library 页面

welm-radio web stop
  停止整个 Web 服务
```

# 8. welm-cdp 不复制，作为本地 npm 包导入

一开始讨论过：当前项目需要 CDP 能力，是不是应该把 `welm-cdp` 代码复制进来？

最后确定：

```txt
不要复制。
不要急着公开发布 npm。
用本地 npm 包 file:../welm-cdp 引入。
```

`welm-radio/package.json`：

```json
{
  "dependencies": {
    "welm-cdp": "file:../welm-cdp"
  }
}
```

`welm-cdp/package.json` 配置 `exports`：

```json
{
  "exports": {
    "./chrome": "./cdp/chrome.js",
    "./client": "./cdp/client.js",
    "./dom": "./cdp/dom.js",
    "./input": "./cdp/input.js",
    "./mouse": "./cdp/mouse.js",
    "./screenshot": "./cdp/screenshot.js"
  }
}
```

关键理解：

```txt
exports 是对外入口白名单。
包内部相对 import 不受 exports 限制。

dom.js 内部可以 import "./client.js"。
外部是否能 import "welm-cdp/client" 取决于 exports。
```

后来因为 `cmd.js` 中用了：

```js
import { closeAllClients } from "welm-cdp/client";
```

所以 `./client` 需要出现在 `welm-cdp` 的 `exports` 中。

安装流程：

```bash
cd welm-radio
npm install
npm install -g .
```

区别：

```txt
npm install
  安装 dependencies，让 import "welm-cdp/xxx" 能找到

npm install -g .
  安装 welm-radio 全局命令
```

# 9. CDP 页面等待逻辑修正

在实际运行：

```bash
welm-radio audiomanager ready --verbose
```

时发现页面肉眼已经打开，但终端卡住，最后报：

```txt
Page load timeout
```

定位到 `welm-cdp/chrome.js` 中 `waitPageReady()` 依赖：

```js
Page.loadEventFired
```

但本地页面加载很快，可能在监听之前事件已经过去。

结论：

```txt
不要依赖 Page.loadEventFired。
改为轮询 document.readyState。
```

推荐逻辑：

```js
await waitLeaveAboutBlank(targetId, options);

while (...) {
  const result = await client.Runtime.evaluate({
    expression: "document.readyState",
    returnByValue: true,
  });

  const readyState = result.result?.value;

  if (readyState === "interactive" || readyState === "complete") {
    return;
  }
}
```

同时讨论了：

```js
await client.Page.enable();
await client.Runtime.enable();
```

结论：

```txt
如果不监听 Page 事件，Page.enable() 可以省略。
Runtime.evaluate 通常也不必显式 Runtime.enable()。
```

# 10. Express 基础认知

今天梳理了 Express 的几个关键点。

## 静态目录

```js
app.use(express.static(path.join(__dirname, "ui")));
```

表示把 `ui` 目录挂到网站根路径。

多个静态目录可以：

```js
app.use("/", express.static(path.join(__dirname, "ui")));
app.use("/assets", express.static(path.join(__dirname, "assets")));
```

访问：

```txt
/audio-library/
 /audio-library/assets/xxx.js
```

如果挂载：

```js
app.use("/audio-library", express.static(path.join(__dirname, "audio-library/ui")));
```

那么 `ui/index.html` 可以通过：

```txt
http://localhost:3000/audio-library/
```

访问。

## Router

`web/audio-library/router.js` 中保留：

```js
export const router = express.Router();
```

不改成：

```js
audioLibraryRouter
```

原因：

```txt
router 是子系统自己的通用出口。
业务语义只在 web/server.js 装配时赋予。
```

在 `web/server.js` 中使用 import alias：

```js
import { router as audioLibraryRouter } from "./audio-library/router.js";
```

这是最小侵入、最干净的方式。

# 11. Audio Library service 层

`web/audio-library/service/` 下目前拆出：

```txt
audio.js
file.js
meta.js
```

## file.js

`scanFiles(dir, options)` 递归扫描文件。

讨论过：

```js
fs.readdirSync(dir, { withFileTypes: true })
```

含义：

```txt
读取当前目录条目，返回 Dirent 对象。
可以判断 item.isFile() / item.isDirectory()。
不会自动递归，需要自己递归。
```

发现并修正过的 bug：

```txt
scanFiles() 最后必须 return result。
递归时要传 options：
scanFiles(fullPath, options)
```

返回对象建议：

```js
{
  path: "/Users/xxx/Music/song.mp3",
  root: "/",
  dir: "/Users/xxx/Music",
  base: "song.mp3",
  ext: ".mp3",
  name: "song"
}
```

注意：曾经在 `audio.js` 中解构：

```js
({ base, name, dir, path })
```

这会和 Node 的 `path` 模块冲突，所以建议改名：

```js
({ base, name, dir, path: filePath })
```

## audio.js

职责：

```txt
1. 从 config 读取 radio.audio_library
2. scanFiles 扫描音频文件
3. 为每个音频计算 metaPath
4. loadMeta
5. 返回前端列表
```

音频扩展名：

```js
[".mp3", ".flac", ".wav", ".m4a", ".aac", ".ogg"]
```

如果目录未配置，建议返回：

```js
[]
```

而不是：

```js
null
```

因为前端 table 更好处理。

# 12. meta 结构演化

这是今天很重要的一条线。

最初 meta 只是音频描述文件，后来逐渐和电台播放逻辑结合。

讨论过但最终去掉的字段：

```txt
artist   偏传统音乐系统，不适合广播编排系统
album    同上
enabled  属于运行时策略，不属于音频描述本身
description 和 content 重复，保留 content
segments 维护成本太高，去掉
```

最终当前阶段推荐 meta：

```json
{
  "title": "",
  "type": "",
  "content": "",
  "tags": [],
  "mood": [],
  "duration": null,
  "start": 0,
  "end": null,
  "cutPoints": [],
  "createdAt": "",
  "updatedAt": ""
}
```

字段含义：

```txt
title
  标题

type
  音频类型，例如 music / voice / effect / intro / outro / weather / news

content
  音频内容描述，给人和 AI 用。
  帮助 AI 选择更合适的音频。

tags
  普通标签

mood
  氛围、情绪标签

duration
  原始音频总时长，单位秒

start
  默认播放起点。
  用于跳过开头空白、杂音、未进入主节奏的部分。

end
  默认播放终点。
  null 表示播放到 duration。
  用于跳过结尾拖尾、空白、噪声。

cutPoints
  可安全切断点，单位秒。
  用于插播时选择自然切点。

createdAt / updatedAt
  创建和更新时间，格式：
  2026-07-02T00:00:00.000Z
```

## 关于 segments

一开始想用类似字幕的结构：

```json
segments: [
  {
    "start": 0,
    "end": 10,
    "text": ""
  }
]
```

后来重新理解实际需求：

```txt
segments 要维护每个音频的完整段落信息，太痛苦。
现实中更容易标几个可切点。
```

所以最终不使用 `segments`，改为：

```json
"cutPoints": [22.8, 51.6, 78.2]
```

而不是复杂对象：

```json
{
  "time": 22.8,
  "type": "sentence-end",
  "priority": 80,
  "text": ""
}
```

理由：

```txt
当前阶段只需要知道哪里能切。
数字数组维护成本最低。
```

## start/end 与 cutPoints 的区别

```txt
start / end
  定义默认可播放范围。

cutPoints
  定义播放中如需插播，哪些点适合切断。
```

例如音频 100 秒：

```json
{
  "duration": 100,
  "start": 3,
  "end": 92,
  "cutPoints": [22.8, 51.6, 78.2]
}
```

默认播放：

```txt
3 -> 92
```

如果播放到 40 秒时需要插播，找下一个 cutPoint：

```txt
51.6
```

队列变成：

```txt
A: 3 -> 51.6
插播节点
A: 51.6 -> 92
```

如果没有 cutPoint：

```txt
当前音频播到 end 后再插播。
```

# 13. meta.js 设计

`createDefaultMeta()`：

```js
function createDefaultMeta() {
  return {
    title: "",
    type: "",
    content: "",
    tags: [],
    mood: [],

    duration: null,

    start: 0,
    end: null,

    cutPoints: [],

    createdAt: "",
    updatedAt: "",
  };
}
```

讨论过是否需要 `normalizeMeta()`。

结论：

```txt
不需要单独 normalizeMeta。
```

因为：

```txt
meta 文件由程序创建。
前端表单展示时把数组转字符串。
保存时再把字符串转数组。
loadMeta 只负责读取，不负责复杂类型修正。
```

但 `loadMeta()` 可以在读取时简单补默认字段：

```js
return {
  ...createDefaultMeta(),
  ...JSON.parse(fs.readFileSync(metaPath, "utf-8")),
};
```

`saveMeta()`：

```js
export function saveMeta(metaPath, data = {}) {
  const oldMeta = fs.existsSync(metaPath) ? loadMeta(metaPath) : null;
  const now = new Date().toISOString();

  const meta = {
    ...createDefaultMeta(),
    ...oldMeta,
    ...data,
    createdAt: oldMeta?.createdAt || now,
    updatedAt: now,
  };

  fs.writeFileSync(metaPath, JSON.stringify(meta, null, 2), "utf-8");

  return meta;
}
```

关于时间格式：

```js
new Date().toISOString()
```

会生成：

```txt
2026-07-02T00:00:00.000Z
```

这是正确格式。

# 14. API 返回中需要路径，但 meta 文件不要写路径

讨论过是否 meta 中需要音频文件路径。

结论：

```txt
meta 文件中不要写 filePath。
```

原因：

```txt
meta 文件和音频文件放在一起。
路径可以由扫描目录动态生成。
如果把绝对路径写进 meta，将来移动音频目录会失效。
```

所以区分：

```txt
.meta.json
  只保存音频描述信息。

API 返回对象
  可以包含 path / metaPath / base / title / content 等运行时信息。
```

例如 API 返回：

```js
{
  base: "Night Jazz.mp3",
  path: "/Users/xxx/Music/Night Jazz.mp3",
  metaPath: "/Users/xxx/Music/Night Jazz.meta.json",
  title: "",
  type: "",
  content: "",
  tags: [],
  mood: [],
  duration: null,
  start: 0,
  end: null,
  cutPoints: []
}
```

# 15. 项目结构比代码更难

最后总结出一个非常重要的认知：

```txt
代码本身通常不是最头疼的。
最头疼的是项目结构。
```

真正消耗精力的问题是：

```txt
这个文件放哪？
这个模块归谁管？
这个名字以后会不会别扭？
这个东西是 infra、web、cmd，还是子系统内部？
```

今天最重要的成果就是边界收敛：

```txt
welm-cdp
  Chrome/CDP 能力

cmd/web.js
  Web 相关 CLI 命令

web/runner.js
  Web 服务启动/停止

web/server.js
  Web HTTP 服务入口

web/audio-library
  Audio Library 子系统

web/audio-library/service
  Audio Library 业务逻辑

web/audio-library/router.js
  Audio Library API 路由

web/audio-library/ui
  Audio Library 页面
```

这套结构已经比最初的 `audiomanager/server.js + ui + core` 稳定很多。

可以用这条线概括：
