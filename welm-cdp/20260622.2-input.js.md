# 一、整体检查结论（重要）

## 1. 架构层面：是“对的”

你现在这层 input.js 已经具备三个清晰分层：

```txt
CDP 原始能力
    ↓
语义封装（click/type/fill）
    ↓
行为组合（clickAt/doubleClickAt/press）
```

这个结构是稳定的，没有明显方向性问题。

---

## 2. 真正的问题不在功能，而在“自动化伪装层缺失”

你现在的 input.js：

```txt
能模拟用户操作
但不能“像人”
```

差距主要在：

```txt
缺少 human-like interaction model
```

也就是：

```txt
节奏
随机性
轨迹
延迟
输入节奏
行为扰动
```

# 二、反自动化检测的核心点（你现在缺的）

现代网页反自动化主要检测 4 类行为：

---

## ① Mouse move 是否“过于机械”

你现在：

```js
mouseMoveTo → 直接 dispatch mouseMoved
```

问题：

```txt
直接跳点
无轨迹
无速度变化
```

### ❌ 容易被检测：

- pointer velocity 恒定
    
- 直接瞬移
    
- click 没有 hover path
    

---

### ✔ 建议增强：轨迹模拟

新增：

```js
moveMouseHumanLike()
```

逻辑：

```txt
A → B
拆成多个点
加入 easing + jitter
```

例如：

```js
for (t = 0; t < 1; t += 0.05) {
  x = lerp(x1, x2, ease(t)) + random(-1,1)
  y = lerp(y1, y2, ease(t)) + random(-1,1)
}
```

---

## ② click 太“干净”

你现在 click：

```txt
move → down → up（固定顺序 + 固定时间）
```

### ❌ 问题：

真实用户：

```txt
停顿
轻微抖动
hover delay
down/up 不对称
```

---

### ✔ 建议增强：

```js
await randomDelay(30~120ms)  // hover pause
await mouseDown
await randomDelay(20~80ms)
await mouseUp
```

---

## ③ typing（最关键的反检测点）

你现在 type：

```js
Input.insertText({ text })
```

### ❌ 这是“高风险点”

因为：

```txt
一次性注入字符串
```

反自动化检测非常容易识别：

- 没有 keydown/keyup
    
- 没有打字间隔
    
- 没有修正行为
    

---

### ✔ 建议改造（强烈建议）

增加：

```js
typeHuman()
```

拆成：

```txt
keydown → keypress → keyup
+ per-character delay
+ random pause
+ occasional backspace
```

例如：

```js
for (char of text) {
  keyDown(char)
  keyUp(char)
  sleep(30~180ms random)
}
```

---

### ⭐ 进阶（非常重要）

加入“人类错误模型”：

```txt
1% 概率打错字
然后 backspace 修正
```

这类行为是：

👉 反 bot detection 的关键特征之一

---

## ④ focus + selection 行为过于理想化

你现在：

```js
setSelectionRange(len, len)
```

或者：

```js
selectNodeContents
```

### ❌ 问题：

真实用户：

```txt
不会总是精确到 end
```

---

### ✔ 建议：

随机策略：

```txt
start selection at random position
or no selection at all
```

---

# 三、input.js 本身的结构问题（轻微）

## 1. “能力层 + 行为层混在一起”

现在：

```txt
click(selector)
clickAt(x,y)
mouseDownAt
mouseUpAt
```

问题：

```txt
行为组合逻辑散落
```

---

### ✔ 建议分层（符合你现在 cdp 收敛风格）

```txt
input/raw/
  mouseDown
  mouseUp
  keyDown
  keyUp

input/behavior/
  click
  doubleClick
  type
  fill
```

---

## 2. buildElementResolver 太“万能”

问题：

```js
nth + selector = JS injection string
```

风险：

- 可读性下降
    
- debug 难
    
- SSRF-like injection 风险（虽然本地）
    

---

建议：

```txt
拆成 DOM resolver function
而不是 string template
```
