# welm-cdp 设计演化记录（2026-06-25，Input 模型重构）

今天是 **welm-cdp** 设计过程中一个重要的转折点。

这次不是代码优化，而是**对 Chromium Input 模型重新认识**。经过大量实验，几乎推翻了之前以 Keyboard 为中心的设计，重新建立了以 **Character / Edit / Keyboard** 为中心的输入体系。

---

# 一、放弃「模拟键盘」，开始研究 Chromium Input

过去一直认为：

```text
Keyboard
    ↓
所有输入
```

因此设计了：

```
KEY_MAP
↓
buildKeyEvent
↓
pressKey
↓
typeText
```

维护：

- key
    
- code
    
- windowsVirtualKeyCode
    
- nativeVirtualKeyCode
    
- modifiers
    

经过今天大量实验，发现这是错误的抽象。

真正的 Chromium Input 并不是 Keyboard。

---

# 二、dispatchKeyEvent(text) 的重大突破

今天最大的发现：

```
dispatchKeyEvent({
    type: "keyDown",
    text: "a",
})
```

会产生：

```
keydown
keypress
beforeinput
input
```

随后：

```
keyUp
```

完成整个输入流程。

进一步实验：

```
text: "你好"
```

同样完全正常。

说明：

> text 并不是一个字符，而是一段待输入的 Unicode 文本。

它可以是：

```
"a"
"Hello"
"你好"
"Hello 世界"
```

并且都会触发完整输入事件。

---

# 三、Enter 的真正实现

原本一直认为：

```
Enter
↓
key + code + windowsVirtualKeyCode
```

今天实验发现：

真正让浏览器执行回车的是：

```
text: "\r"
```

而不是：

```
key: "Enter"
```

也就是说：

```
Enter
≈
"\r"
```

这彻底改变了 Enter 的实现方式。

---

# 四、insertText 的重新定位

以前准备大量使用：

```
Input.insertText()
```

后来实验发现：

它只产生：

```
beforeinput
input
```

完全不会产生：

```
keydown
keypress
keyup
```

因此：

它属于：

```
Input Layer
```

而不是：

```
Keyboard Layer
```

最终决定：

普通输入不再优先使用 insertText。

---

# 五、发现 commands 是 Chromium 编辑命令

最开始：

```
Cmd+V
```

一直模拟快捷键。

后来发现：

```
commands:["Paste"]
```

即可完成真正粘贴。

继续实验：

```
commands:["SelectAll"]
commands:["DeleteBackward"]
```

均成功。

同时测试页面得到：

```
beforeinput
input
```

并且：

```
inputType
```

正确变成：

例如：

```
deleteContentBackward
```

说明：

commands 根本不是 Keyboard。

而是：

```
Chromium Editing Command
```

---

# 六、Input 三层模型正式形成

今天最终形成的新模型：

```
Character
──────────────
dispatchKeyEvent(text)

Edit
──────────────
commands

Keyboard
──────────────
少量控制键
```

以后：

字符：

```
text
```

编辑：

```
commands
```

控制键：

```
key
```

彻底分离。

---

# 七、KEY_MAP 开始失去意义

今天最大的架构变化：

决定逐步删除：

```
KEY_MAP
```

原因：

普通字符：

```
text
```

即可。

完全不需要：

```
KeyA
Digit1
KeyB
...
```

以后只保留真正控制键即可。

---

# 八、buildKeyEvent 被淘汰

同样决定删除：

```
buildKeyEvent()
```

原因：

已经不存在统一的 Keyboard Event。

现在：

Character

Edit

Keyboard

完全是三条不同路径。

统一构造器没有意义。

---

# 九、normalizeKeyEvent 被放弃

曾讨论：

```
normalizeKeyEvent()
```

后来也决定不要。

理由：

用户更喜欢：

每一个能力完全独立。

例如：

```
Enter
```

自己写：

```
key
code
windowsVirtualKeyCode
```

而不是：

通过 normalize 自动补。

整个 input.js 更直观。

---

# 十、重新思考 pressKey

经过实验：

发现：

```
text
```

才是真正负责字符输入。

因此：

```
pressKey("a")
```

已经没有太大意义。

以后：

字符：

```
dispatchKeyEvent(text)
```

控制键：

```
pressKey()
```

职责彻底分开。

---

# 十一、重新设计 typeText

今天最终倾向：

输入：

前若干字符：

```
dispatchKeyEvent(text)
```

一个一个输入。

剩余文本：

```
commands:["Paste"]
```

一次完成。

原因：

既有完整事件：

```
keydown
keypress
beforeinput
input
keyup
```

又不会大量逐字符输入。

比：

```
insertText
```

更接近真人。

也比：

```
pressKey
```

简单。

---

# 十二、重新思考 makeTypo

以前：

makeTypo

主要为了补：

```
keydown
keypress
keyup
```

现在：

dispatchKeyEvent(text)

已经天然具备完整事件。

因此：

makeTypo 的意义开始下降。

是否保留，

准备后续再决定。

---

# 十三、CLI 参数新增数组字面量

cmd-resolver 增加：

```
[aaa,bbb,ccc]
```

自动解析：

```
["aaa","bbb","ccc"]
```

随后升级：

数组内部继续调用：

```
parseValue()
```

因此：

```
[1,true,abc]
```

自动变成：

```
[
    1,
    true,
    "abc"
]
```

实现保持：

```
parseValue()
```

作为唯一解析来源。

避免重复逻辑。

---

# 十四、commands API

新增能力：

```
editCommand(targetId, command)
```

支持：

```
"Paste"
```

以及：

```
[
    "SelectAll",
    "DeleteBackward"
]
```

内部统一：

```
Array.isArray()
```

转换。

讨论认为：

以后：

不需要：

```
selectAll()
paste()
deleteBackward()
```

全部封装。

直接：

```
editCommand()
```

即可。

只有真正业务组合：

例如：

```
clearAll()
```

再封装。

符合：

```
能力
↓

组合
↓

业务
```

一直坚持的设计思想。

---

# 十五、整个 Input 模型最终演化

几天前：

```
Keyboard
↓

一切
```

今天：

```
Character
    ↓
dispatchKeyEvent(text)

Edit
    ↓
commands

Keyboard
    ↓
少量控制键
```

这是整个 **welm-cdp Input** 最重要的一次架构升级。

不再围绕 Keyboard，

而是围绕 Chromium Input 自身设计。
