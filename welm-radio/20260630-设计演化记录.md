# 📻 welm-radio & config/store 设计演化记录（当前会话总结）

## 🧭 1. 项目背景（最初目标）

用户正在设计一个本地 AI 电台系统：

```text
welm-radio = 本地 AI 电台系统
```

- music（音乐）
    
- voice（主播/语音）
    
- news（新闻）

- AI 编排（playlist / graph）
    
- Web Audio API 播放
    
- Chrome/CDP 作为播放器容器
    

---

## 🧱 2. 系统架构共识（radio 分层）

最终稳定分层：

```text
AI → Playlist/Graph → Engine → Player(Web Audio)
```

### ✔ 关键原则

- AI 不直接控制播放
    
- Engine 只执行 graph
    
- Player 只做音频输出
    
- 数据与执行严格分离
    

---

## 🎛️ 3. broadcast-engine 设计演化

最终稳定认知：

### ✔ engine 定位

```text
broadcast-engine = 调度器 (scheduler)
```

职责：

- 只消费 broadcastGraph
    
- 维护 current/next node
    
- 负责 getNext / step
    
- 不参与数据生成
    

---

## 📦 4. media & broadcast 数据模型

### media-registry

```text
媒体索引（文件 + tags + duration）
```

### broadcast.js

```text
节目图（node + edges + selector）
```

### 关键原则

- engine 不生成数据
    
- engine 只消费“已准备好的结构”
    

---

## 🌐 5. mediaRoot 设计演化（关键讨论）

### 初始问题

- 是否每次运行 setMediaRoot？
    
- bootstrap 是否只执行一次？
    
- config 如何持久化？
    

---

### ❌ 错误理解阶段

- 把 bootstrap 当“永久设置”
    
- 误认为每次 play 要初始化 mediaRoot
    

---

### ✔ 最终正确模型

```text
init（一次） → 写 config.json
run（每次） → 读取 config.json
```

---

### ✔ 关键结论

- bootstrap ≠ 永久配置
    
- bootstrap = 当前进程初始化
    
- 持久化必须落 disk
    

---

## 💾 6. config 系统演化（核心）

### 初始设计

```js
config.getConfigValue()
config.setConfigValue()
```

问题：

- 命名冗长
    
- API 语义不统一
    
- 容易误解为对象操作
    

---

### ➜ path access 设计

```js
get("a.b.c")
set("a.b.c", value)
```

---

### ➜ setByPath / getByPath 引入

形成：

```text
deep KV store
```

---

## ⚠️ 7. 最大语义冲突问题

用户关键洞察：

> get/set 让人误以为是操作对象，而不是持久化存储

---

## 🔁 8. 命名演化路径

### config → configStore → store → config

过程：

1. config（最初）
    
2. configStore（语义更清晰）
    
3. store（更抽象）
    
4. 回归 config（用户偏好）
    

---

## ✔ 9. 当前最终选择

用户最终确定：

```js
export const config = {
  load,
  get,
  set,
};
```

### ✔ 理解方式

```text
config = persistent configuration API
```

而不是：

```text
config = object
```

## 🧠 11. 关键架构共识（已稳定）

### ✔ store vs config

- config.json = 数据文件
    
- config module = API
    
- store = 行为模型（KV abstraction）
    

---

### ✔ domain separation

未来结构：

```text
infra/config
media/domain
radio/domain
engine/core
```

---

## ⚖️ 12. 核心设计原则（最终共识）

### ✔ 1. 语义分层

```text
infra → domain → engine
```

---

### ✔ 2. config 不能暴露 object access

必须：

```js
config.get("a.b.c")
```

禁止：

```js
config.a.b.c
```

---

### ✔ 3. 持久化 vs runtime 分离

- config.json = persistent
    
- engine state = runtime
