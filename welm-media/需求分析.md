
> [!NOTE]
> 你知道深夜电台吗，一个播主对听众讲述一些主题内容，背景有不同的音乐播放，播主说话时背景影月声音渐渐降低，不说的时候听歌时候音乐声音渐渐升高，就是有很多条音频叠加播放
> 如果我本地有很多mp3，有音乐，有段子，有天气，有新闻等等，我想通过node项目，模拟电台主播，请问是否可行，有哪些可行性方案？

可行，而且很适合做成你的 **welm-media** 里的一个模块。

你描述的其实是一个“自动编排音频节目”的系统，核心不是简单播放 MP3，而是：

**多条音频轨道叠加 + 主播语音优先 + 背景音乐自动压低 + 空白时音乐抬高 + 按节目单顺序生成最终音频。**

专业上这类效果叫：

- **音频混音**：多条音频叠加成一条输出
    
- **ducking / sidechain compression**：人声出现时，背景音乐自动降低
    
- **crossfade**：音乐之间平滑过渡
    
- **playlist / rundown**：电台节目单
    
- **voiceover**：主播旁白

## 方案一：最现实，Node 负责调度，FFmpeg 负责混音

这是最推荐的方案。

Node 不直接处理音频波形，而是负责：

1. 扫描本地素材  
    例如：
    
    ```text
    music/
    jokes/
    weather/
    news/
    intro/
    outro/
    ```
    
2. 生成节目单  
    例如：
    
    ```js
    [
      { type: "music", file: "night-jazz.mp3", duration: 60 },
      { type: "voice", file: "opening.mp3" },
      { type: "music", file: "lofi.mp3", duration: 120 },
      { type: "voice", file: "weather.mp3" },
      { type: "voice", file: "joke.mp3" },
      { type: "music", file: "ending.mp3", duration: 90 }
    ]
    ```
    
3. 计算每条音频的开始时间、淡入淡出、音量
    
4. 拼出 FFmpeg 命令
    
5. 输出一个完整节目：
    
    ```text
    radio-2026-06-19.mp3
    ```
    

FFmpeg 负责：

- 音量调整
    
- 音频延迟
    
- 淡入淡出
    
- 多轨混合
    
- 背景音乐 ducking
    
- 输出 MP3/AAC/WAV
    

这个架构最稳，因为 FFmpeg 是音视频处理的底层神器，Node 适合做流程编排。

## 方案二：先做“固定电台节目单”，不用实时自动 ducking

第一阶段可以别一上来做复杂 sidechain。

可以先做一个简单规则：

**主播说话时，手动把背景音乐降到 20%；主播结束后，音乐 2 秒内升回 80%。**

例如节目结构：

```text
00:00  开场音乐，音量 80%
00:10  主播开场，音乐 2 秒内降到 20%
00:35  主播结束，音乐 2 秒内升到 80%
01:00  播放段子，音乐降到 15%
01:40  段子结束，音乐升高
02:00  播放新闻
03:00  播放歌曲
```

这个方案不需要检测人声，只要知道每段主播音频的开始和结束时间即可。

优点：

- 容易实现
    
- 效果已经很像深夜电台
    
- 适合 CLI
    
- 适合你现在的 welm-media 风格
    

缺点：

- 如果主播音频里面有停顿，音乐不会自动升高
    
- 不够“智能”
    

但作为第一版，非常合适。

## 方案三：真正模拟电台主播，使用 TTS 生成主播语音

如果你不只是已有 MP3，还想让 Node 自动生成主播内容，就可以加一层：

```text
文本稿件 -> TTS生成主播音频 -> 与音乐混音 -> 输出节目
```

例如本地有：

```text

news.txt
jokes.txt
```

Node 生成稿子：

```text
晚上好，欢迎来到今晚的深夜电台。
现在是盐湖城时间晚上十点。
接下来是一段轻音乐，随后我们聊聊今天的天气。
```

然后用 TTS 生成主播 MP3，再和背景音乐混合。

这一版就更像“AI 电台”。

架构可以是：

```text
素材库
  ↓
节目脚本生成
  ↓
TTS 主播语音
  ↓
FFmpeg 混音
  ↓
最终电台音频
```

## 方案四：实时播放版，不生成文件

还有一种是“实时电台播放器”。

Node 启动后持续播放：

```text
背景音乐一直播放
主播语音插入时音乐降低
主播结束音乐升高
下一首音乐自动衔接
```

这个更像真正电台直播。

但复杂度明显更高，因为你要处理：

- 实时播放设备
    
- 播放队列
    
- 多音轨同步
    
- 中途插播
    
- 暂停、跳过、音量控制
    
- 可能还要做本地 Web UI
    

Node 可以做，但不建议第一版就做。

第一版更建议：**生成一个完整 MP3 文件**，效果确定、调试容易、可反复试听。

第一版不用做太大，只做这些就够：

```text

2. 从 voice 目录按顺序插入几段主播音频
3. 主播出现时音乐降低到 20%
4. 主播结束后音乐恢复到 70%
5. 最终输出一个 MP3
```

这已经能产生“深夜电台”的味道。

## 难点在哪里

主要难点不是 Node，而是音频时间线。

你需要处理：

1. 每个 MP3 的时长  
    可以用 `ffprobe` 获取。
    
2. 多条音频的开始时间  
    每段 voice 应该在第几秒插入。
    
3. 背景音乐长度不够怎么办  
    可以循环、拼接、换下一首。
    
4. 音乐降噪和升高是否自然  
    需要 fade in / fade out。
    
5. 不同 MP3 音量差异大  
    需要 loudness normalization，例如统一到接近的响度。

## 先区分两种目标

你现在讨论的是**实时播放**，不是生成最终 MP3。

### 生成文件模式

```text
多个 mp3 -> ffmpeg 混音 -> radio.mp3 -> 用 foobar 播放
```

优点是简单、稳定。缺点是播放时已经固定，不能实时插播、实时调音。

### 实时播放模式

```text
多个 mp3 -> 程序实时解码 -> 程序实时混音 -> 声卡输出
```

优点是可以随时控制：

```text
背景音乐音量降到 20%
插播主播
暂停新闻
切歌
淡入淡出
临时播放天气
```

缺点是复杂度高一些。

你现在问的就是第二种。

# 可行方案一：Node 控制 FFplay / FFmpeg 子进程

这是最容易起步的方案，但控制能力有限。

例如开一个背景音乐进程：

```bash
ffplay -nodisp -autoexit music.mp3
```

Node 里用：

```js
spawn("ffplay", ["-nodisp", "-autoexit", "music.mp3"]);
```

再开一个主播音频：

```js
spawn("ffplay", ["-nodisp", "-autoexit", "voice.mp3"]);
```

这样确实可以做到“多个播放器同时播放”。

但是问题很明显：

1. 每个进程的音量不好实时控制
    
2. 多个音频流之间不好精确同步
    
3. ducking 不好做
    
4. 跨平台控制麻烦
    
5. 播放器之间没有统一时钟
    
6. fade in / fade out 只能靠额外参数或重新启动进程
    

所以这个方案适合做非常粗糙的 Demo：

```text
播放背景音乐
插入一段语音
大概叠在一起
```

但不适合做你说的深夜电台效果。

结论：**能做，但不是好方案。**

---

# 可行方案二：Node 启动一个 FFmpeg 实时混音管线

这个比方案一专业一些。

思路是：

```text
Node 负责决定什么时候插入什么音频
FFmpeg 负责实时混音
输出到系统音频设备
```

理论上可以用 FFmpeg 的 `amix`、`volume`、`sidechaincompress` 等滤镜做实时混音。

问题是：

FFmpeg 很擅长**一次性生成文件**，但不太适合做一个长期运行、可随时插播、随时改音量的“实时控制台”。

如果节目单是提前确定的，比如：

```text
00:00 播音乐
00:10 插主播
00:40 插天气
01:00 插新闻
```

那 FFmpeg 很合适。

但如果你想运行中随时：

```text
现在插播天气
现在切歌
现在把背景音乐降到 10%
现在恢复到 70%
```

FFmpeg 就会变得很别扭。

结论：**适合“准实时节目单播放”，不适合高度交互控制。**

# 可行方案三：用 Node 音频库，自己做实时混音

这是最符合你需求的方向。

架构大概是：

```text
MP3 文件
  ↓
解码成 PCM 音频数据
  ↓
每条音轨有自己的状态：
  - 当前播放位置
  - 音量
  - 是否暂停
  - 淡入淡出
  - 是否循环
  ↓
混音器把多条 PCM 叠加
  ↓
输出到系统声卡
```

程序内部可以这样理解：

```js
mixer.addTrack("music", "music/night.mp3", {
  volume: 0.7,
  loop: true
});

mixer.addTrack("voice", "voice/opening.mp3", {
  volume: 1.0
});

mixer.fade("music", 0.2, 1500); // 1.5 秒内降到 20%
mixer.play("voice");

mixer.onEnd("voice", () => {
  mixer.fade("music", 0.7, 2000); // 主播结束后 2 秒恢复
});
```

这才是你要的“代码随时控制多个音轨”。

---

## 这种方案的本质

不要把每条音轨交给一个播放器。

而是：

```text
每条音轨只是程序里的一个对象
最终只有一个统一的音频输出
```

类似这样：

```text
Track: background music
Track: host voice
Track: joke
Track: weather
Track: news

           ↓
        Mixer
           ↓
      Audio Output
```

这样才可以统一控制。

# 可行方案四：Electron / Web Audio API

这个方案非常值得考虑。

你虽然做的是 Node 项目，但如果目标是“可编程实时播放”，其实浏览器内置的 **Web Audio API** 很适合。

架构是：

```text
Electron App
  ├─ Node 负责扫描文件、生成节目单
  └─ Chromium Web Audio API 负责播放和混音
```

Web Audio API 天然支持：

- 多音轨播放
    
- GainNode 控制音量
    
- 音量渐变
    
- 多轨混合
    
- 定时播放
    
- 音频分析
    
- 压缩器
    
- 滤镜
    
- 声像控制
    

它的模型非常适合电台：

```text
AudioBufferSourceNode 音源
  ↓
GainNode 音量控制
  ↓
DynamicsCompressorNode 可选压缩
  ↓
AudioDestinationNode 声卡输出
```

人声来了：

```js
musicGain.gain.linearRampToValueAtTime(0.2, audioCtx.currentTime + 1.5);
```

人声结束：

```js
musicGain.gain.linearRampToValueAtTime(0.7, audioCtx.currentTime + 2);
```

这个体验会比纯 Node 音频库更舒服。

结论：如果你愿意做一个小型桌面 App，**Electron + Web Audio API 是非常适合的方案。**

例如：


启动后：


浏览器页面负责真正播放。

Node 负责：

```text

提供文件 URL

```

浏览器负责：


这其实比“纯 Node 输出声卡”更容易实现。

你甚至可以先不做 UI，只打开一个页面，自动播放。

结论：**这是我最推荐的实时播放第一版。**

# 不推荐的方式

你提到的这种：

```text
开多个终端
每个终端播放一条音轨
用代码分别控制它们
```

理论上可以，但不推荐。

因为它会变成：

```text
终端1 -> 播音乐
终端2 -> 播主播
终端3 -> 播天气
终端4 -> 播新闻
```

问题是：

1. 音量控制分散
    
2. 时间同步困难
    
3. 每个进程都有自己的播放状态
    
4. ducking 需要跨进程协调
    
5. 进程崩了不好恢复
    
6. 延迟不可控
    
7. 后续扩展很痛苦
    

这不是“电台混音台”，更像“几个播放器同时乱放”。

# 你真正需要的抽象

你可以把它设计成一个 `RadioPlayer`：

```js
const radio = new RadioPlayer();

await radio.loadTrack("music", "/music/night.mp3", {
  loop: true,
  volume: 0.7
});

await radio.play("music");

await radio.duck("music", {
  to: 0.2,
  duration: 1500
});

await radio.playOnce("voice", "/voice/weather.mp3");

await radio.fade("music", {
  to: 0.7,
  duration: 2000
});
```

也就是说，你的 Node 层不应该关心“开几个播放器”，而应该关心：

```text
音轨
播放
停止
暂停
音量
淡入
淡出
插播
恢复
```

## 页面内部的播放器模型

网页里可以有一个 `RadioPlayer`：

```js
class RadioPlayer {
  constructor() {
    this.ctx = new AudioContext();
    this.tracks = new Map();
    this.masterGain = this.ctx.createGain();
    this.masterGain.connect(this.ctx.destination);
  }

  async load(name, url) {
    const res = await fetch(url);
    const arrayBuffer = await res.arrayBuffer();
    const audioBuffer = await this.ctx.decodeAudioData(arrayBuffer);

    this.tracks.set(name, {
      buffer: audioBuffer,
      source: null,
      gain: this.ctx.createGain(),
      volume: 1,
    });

    this.tracks.get(name).gain.connect(this.masterGain);
  }

  play(name) {
    const track = this.tracks.get(name);
    const source = this.ctx.createBufferSource();

    source.buffer = track.buffer;
    source.connect(track.gain);
    source.start();

    track.source = source;
  }

  fade(name, to, seconds) {
    const track = this.tracks.get(name);
    const now = this.ctx.currentTime;

    track.gain.gain.cancelScheduledValues(now);
    track.gain.gain.setValueAtTime(track.gain.gain.value, now);
    track.gain.gain.linearRampToValueAtTime(to, now + seconds);
  }
}
```

这样你就可以在页面里做：

```js
await radio.load("music", "/music/night.mp3");
await radio.load("voice", "/voice/opening.mp3");

radio.play("music");
radio.fade("music", 0.2, 1.5);
radio.play("voice");
```

这就是“多音轨同时输出”。

它是一个**网页自动化能力项目**。

但 `welm-radio` 的核心价值是：

```text
音频素材管理
节目单生成
Web Audio API 播放
多轨混音
背景音乐 ducking
AI 动态编排
主播串词
TTS
```

它是一个**音频应用 / 电台系统项目**。

两者只是在“打开 Chrome 播放网页”这个点上有交集，但核心领域不同。

所以如果为了复用“打开 Chrome”能力，就把 `radio` 放进 `welm-cdp`，会导致项目职责变脏。

---

## 更好的关系：welm-radio 调用 welm-cdp

可以让 `welm-radio` 成为独立项目，但在需要时调用 `welm-cdp`。

```text
welm-radio
  ├─ 启动本地 radio server
  ├─ 生成播放器网页
  ├─ 提供 mp3 文件
  ├─ 通过 WebSocket 控制播放
  └─ 可选调用 welm-cdp 打开 Chrome 页面

welm-cdp
  └─ 只负责打开 Chrome / 打开 URL / 点击 Start
```

也就是说：

```text
welm-radio 是业务主项目
welm-cdp 是可选工具依赖
```

这个方向更干净。

---

## 第一版甚至不一定需要 welm-cdp

`welm-radio` 自己也可以打开浏览器页面。

Node 里可以做：

```js
import { spawn } from "node:child_process";

spawn("open", ["http://localhost:3456/player"]);
```

macOS 上：

```bash
open http://localhost:3456/player
```

这样就能用默认浏览器打开页面。

如果你想指定 Chrome，也可以：

```bash
open -a "Google Chrome" http://localhost:3456/player
```

所以第一版 `welm-radio` 可以完全不依赖 `welm-cdp`。

---

## 那 welm-cdp 什么时候有用？

只有当你需要这些能力时，再引入它：

```text
自动启动专用 Chrome profile
自动打开播放器页面
自动点击 Start 按钮
自动执行页面 JS
自动截图调试播放器状态
自动检测页面是否加载完成
```

比如浏览器自动播放限制导致必须点按钮：

```text
页面打开后必须点击 Start Radio 才能播放声音
```

这时 `welm-cdp` 很有用。

但这仍然可以作为可选模式：

```bash
welm-radio live ./materials
```

默认只启动服务并打开网页。

```bash
welm-radio live ./materials --cdp
```

使用 `welm-cdp` 自动打开并点击 Start。

## 推荐项目关系

我建议这样：

```text
welm-radio
  不依赖 welm-cdp 也能运行

welm-cdp
  作为外部辅助工具，可被 welm-radio 调用
```

而不是：

```text
welm-cdp
  内置 radio
```

更不是：

```text
welm-radio
  强依赖 welm-cdp
```

这样以后你不会被绑定住。

## 简单打开浏览器就够了

第一版可以只实现：

```js
export function openBrowser(url, options = {}) {
  const app = options.app ?? "Google Chrome";

  if (process.platform === "darwin") {
    return spawn("open", ["-a", app, url], {
      stdio: "ignore",
      detached: true,
    });
  }

  // 后面再补 linux/windows
}
```

这样 `welm-radio` 不是“浏览器自动化项目”，只是“需要打开一个播放页面”。

这非常合理。

```text
radio/browser.js
```

```text
openBySystemBrowser(url)
openByWelmCdp(url)
```

命令层：

```bash
welm-radio live ./materials --cdp
```

```text
openBySystemBrowser
```

需要自动点击时：

```text
openByWelmCdp
```

这样既独立，又能复用你已有能力。

## 最终判断

你的判断是对的：

```text
welm-radio 应该另开项目。
```

原因很简单：

```text
welm-cdp 的主语是“网页自动化”；
welm-radio 的主语是“电台播放与节目编排”。
```

Chrome 只是 `welm-radio` 的播放容器，不是它的核心业务。

所以最佳路线是：

```text
新建 welm-radio
使用 Node 启动本地服务
使用网页 Web Audio API 做播放器
默认用系统命令打开 Chrome
必要时可选调用 welm-cdp
```

这比把 radio 放进 `welm-cdp` 更干净，也更符合你一直坚持的“能力和业务分开”的设计思路。
